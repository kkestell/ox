# Wire Ur.Tui to real Ur backend, remove Dummy

## Goal

Replace all `Ur.Tui.Dummy` types with a clean interface boundary (`IChatBackend` / `IChatSession`) backed by real `Ur` types in production. Delete the `Dummy/` directory entirely.

## Desired outcome

- `Ur.Tui` defines `IChatBackend` and `IChatSession` — the narrow contract ChatApp needs.
- A production adapter (`UrChatBackend`) wraps `UrHost` behind that interface.
- `Program.cs` boots a real `UrHost`, wraps it in `UrChatBackend`, and the first-run flow works end-to-end.
- `Ur.Tui/Dummy/` is deleted (5 files).
- Tests use a lightweight `TestChatBackend` — no temp dirs, no cache seeding, no `InternalsVisibleTo`.

## How we got here

The Dummy types were scaffolding — parallel versions of `UrConfiguration`, `UrSession`, `AgentLoopEvent`, etc. that let the TUI be developed independently. Now that both sides are stable, we can replace them with a proper interface boundary backed by the real core.

## Approaches considered

### Option A — Direct wiring

- **Summary:** ChatApp takes `UrHost` directly. All Dummy references become real Ur types.
- **Pros:** No new abstractions.
- **Cons:** ChatApp couples to `UrHost`'s full surface area (session persistence, workspace, model catalog caching) when it only needs a narrow slice. Tests require heavy setup: temp directories, seeded model cache files, `InternalsVisibleTo` across projects, internal `UrHost.Start` overloads.
- **Failure modes:** If `UrHost.Start` side effects grow, test setup becomes fragile. Async bridging (`.GetAwaiter().GetResult()`) leaks into ChatApp instead of being handled at the boundary.

### Option B — Interface boundary

- **Summary:** Define `IChatBackend` and `IChatSession` in `Ur.Tui` capturing what ChatApp actually needs. Production adapter wraps `UrHost`. Tests use a simple in-memory implementation.
- **Pros:** ChatApp's contract is explicit and narrow. Tests stay lightweight (no file system, no UrHost construction). Async bridging is handled in the adapter, not in ChatApp. No changes needed to the `Ur` project.
- **Cons:** One more layer. But it's a thin, purpose-driven boundary — not a speculative abstraction.
- **Failure modes:** Interface drift if Ur's API changes significantly. Low risk given the interface is narrow.

## Recommended approach

**Option B — Interface boundary.**

- **Why:** The Dummy types weren't wrong to exist — they were the wrong *shape*. They duplicated Ur's types 1:1 instead of defining what the TUI actually needs. A proper interface captures the same boundary but with clear intent. It keeps test setup trivial, keeps async concerns out of the render loop, and doesn't require cracking open `Ur`'s internals for the TUI test project.
- **Key tradeoffs:** One thin adapter layer in exchange for clean separation, simple tests, and zero changes to the `Ur` project.

## Interface design

`Ur.Tui` still references `Ur` for shared domain types (`AgentLoopEvent`, `ModelInfo`, `UrChatReadiness`, `UrChatBlockingIssue`). These are the shared language between the core and the UI — abstracting them would be pointless. The interface narrows the *orchestration* surface, not the *domain* types.

```csharp
// What ChatApp needs from the backend
public interface IChatBackend
{
    UrChatReadiness Readiness { get; }
    IReadOnlyList<ModelInfo> AvailableModels { get; }
    Task SetApiKeyAsync(string apiKey);
    Task SetSelectedModelAsync(string modelId);
    IChatSession CreateSession();
}

public interface IChatSession
{
    IAsyncEnumerable<AgentLoopEvent> RunTurnAsync(string input, CancellationToken ct);
}
```

Key design choices:
- **Async `SetApiKeyAsync` / `SetSelectedModelAsync`:** These are genuinely async operations (keyring writes, settings file persistence). The interface reflects that. ChatApp handles them the same way it handles `RunTurnAsync` — fire off the work via `Task.Run`, enqueue a completion event, process it on the next frame. This is consistent with the existing event queue pattern in `SubmitMessage()`.
- **`CreateSession()` returns `IChatSession`:** ChatApp doesn't need `UrSession`'s full API (persistence, message history, etc.). It just needs to run turns.
- **Domain types used directly:** `AgentLoopEvent`, `ModelInfo`, `UrChatReadiness`, `UrChatBlockingIssue` — these are small, stable types that both sides need. No wrappers.

### Making the frame callback async

`RenderLoop.RunAsync` is already async. The frame callback is sync (`Func<ReadOnlySpan<KeyEvent>, bool>`) only because `ReadOnlySpan` can't cross async boundaries. But the span comes from a `List<KeyEvent>` (via `CollectionsMarshal.AsSpan`), and `List<T>` implements `IReadOnlyList<T>`. Fix: change the callback to `Func<IReadOnlyList<KeyEvent>, ValueTask<bool>>` and pass the list directly. Zero allocation, fully async.

This makes `ChatApp.ProcessFrame` async, so it can `await` config operations directly — no `Task.Run`, no event queuing for config ops, no bridging gymnastics.

## Structural considerations

**Hierarchy:** `Ur.Tui -> Ur` (for domain types) and `Ur.Tui -> Ur.Terminal` (for rendering). ChatApp depends on interfaces defined in its own project, not on `UrHost` directly. Clean dependency direction.

**Abstraction:** The Dummy types mirrored Ur 1:1 — wrong level of abstraction. `IChatBackend` captures what the TUI *actually needs*, which is a smaller surface. The adapter handles the translation.

**Modularization:** `Ur.Tui/Dummy/` was a mock backend living inside the UI project. Replacing it with an interface + adapter separates the "what I need" (interface) from the "how it's provided" (adapter). The interface is the module boundary.

**Encapsulation:** `UrHost`'s internals (keyring, settings persistence, model catalog caching, session store) are fully hidden behind `UrChatBackend`. ChatApp never sees them.

- **Code paths affected:** `Ur.Terminal/App/RenderLoop.cs` (callback signature), `Ur.Tui/ChatApp.cs` (rewired to interface, now async), `Ur.Tui/Components/ModelPickerModal.cs` (type swap), `Ur.Tui/Program.cs` (startup), all TUI tests.

- **Deleted files:** 5 Dummy files + directory.
- **Dependency impact:** `Ur.Tui` gains a dependency on `Ur` (and transitively on `Microsoft.Extensions.AI`, etc.). Expected — TUI is the frontend for the core.
- **Ur project impact:** None. Zero changes.
- **Ur.Terminal impact:** One method signature change in `RenderLoop` (`Span` -> `IReadOnlyList`, sync -> async callback).

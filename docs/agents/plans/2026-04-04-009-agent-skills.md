# Agent Skills

## Goal

Add a skills system that lets users and the model extend the agent's capabilities through reusable prompt templates defined as `SKILL.md` files. Skills are automatically discovered from `~/.ur/skills/` (user-scoped) and `$WORKSPACE/.ur/skills/` (project-scoped), and can be invoked two ways: users type `/skill-name` as a slash command, or the model calls the built-in `skill` tool.

## Desired outcome

- A `SKILL.md` file dropped into `~/.ur/skills/my-skill/SKILL.md` or `.ur/skills/my-skill/SKILL.md` is automatically discovered on startup and available for invocation.
- Users can type `/my-skill some args` in the TUI or CLI and the skill's expanded prompt replaces their input.
- The model sees available skills listed in a system message and can invoke them via the `skill` tool.
- Argument substitution (`$ARGUMENTS`, named args, `${UR_SKILL_DIR}`, `${UR_SESSION_ID}`) works in skill content.
- Skills with `user-invocable: false` are hidden from slash commands. Skills with `disable-model-invocation: true` cannot be called by the model.

## How we got here

The reference analysis (`/home/kyle/src/research/analysis/claude-skills.md`) describes a mature skills system with many features. We scoped this plan to the core value proposition: directory-based SKILL.md discovery, user slash commands, and model invocation via a tool. Deferred features (marked with TODOs in code):

- **Conditional skills** (`paths` frontmatter) -- parse and store but don't activate
- **Shell execution** (`!`command``) in skill content
- **Allowed-tools / model override** -- parse but don't enforce
- **Forked execution** (`context: fork`) -- parse but always run inline
- **MCP skills, bundled skills, caching, dynamic discovery, listing budgets** -- out of scope entirely

System message injection was chosen over tool-description-only because it's extensible -- other subsystems can contribute to the system prompt later.

## Current state

- **No system prompt mechanism.** The agent loop sends conversation messages + tool definitions to the LLM with no system message. This plan introduces one.
- **No skill/command concept.** Tools are the only extension point for the model. Skills add a second, prompt-template-based extension point.
- **Extension system exists** but is fundamentally different (Lua code registering tools). Skills are markdown prompt templates -- a peer subsystem, not a layer on extensions.
- **YAML is not used anywhere.** The SKILL.md frontmatter parser needs either a YAML library or a simple key-value parser.

## Structural considerations

**Hierarchy**: Skills sit as a peer to Extensions under UrHost. Both are loaded at startup, both contribute to the agent's capabilities, but through different mechanisms (prompts vs. tools). The dependency flows correctly: UrHost -> SkillRegistry -> SkillLoader -> SkillDefinition.

**Abstraction**: Skills are pure data (parsed markdown with metadata). The `SkillTool` is the only piece that bridges skills into the tool system. This keeps the abstraction clean -- skills don't pretend to be tools.

**Modularization**: New `Ur/Skills/` namespace with 4-5 classes is right-sized. Not a nano-module (there's real parsing and expansion logic), not a god module.

**Encapsulation**: The SkillRegistry is the public API surface. Internal details (frontmatter parsing, directory scanning) stay internal. UrSession needs read access to the registry for slash command lookup and system prompt building.

## Refactoring

### Add system prompt support to AgentLoop

Before the feature work, `AgentLoop.RunTurnAsync` needs to accept an optional system prompt string. The loop will prepend a `ChatRole.System` message to the messages it sends to the LLM *without* adding it to the mutable conversation history (so it's not persisted to disk). This is a small, clean change that unblocks both skills and any future system prompt needs.

**What it achieves**: Separates the transient system context (skill listings, instructions) from the persistent conversation history.

## Implementation plan

### Phase 1: Data model and frontmatter parsing

- [x] **Add YamlDotNet package** to `src/Ur/Ur.csproj`. This handles YAML frontmatter robustly (multiline values, special characters) without us writing a fragile custom parser.

- [x] **Create `src/Ur/Skills/SkillDefinition.cs`** -- the core model for a loaded skill.
  ```csharp
  // Key fields from SKILL.md frontmatter + the raw content body.
  // Mirrors the reference Command type, scoped to what we support.
  public sealed class SkillDefinition
  {
      public required string Name { get; init; }
      public required string Description { get; init; }
      public string? WhenToUse { get; init; }
      public bool UserInvocable { get; init; } = true;
      public bool DisableModelInvocation { get; init; }
      public string? ArgumentHint { get; init; }
      public string[]? ArgumentNames { get; init; }
      public string? Context { get; init; }        // "inline" | "fork" -- TODO: fork support
      public string? Agent { get; init; }           // TODO: agent type for forked execution
      public string[]? Paths { get; init; }         // TODO: conditional activation
      public string[]? AllowedTools { get; init; }  // TODO: tool restriction
      public string? Model { get; init; }           // TODO: model override
      public string? Version { get; init; }
      public required string Content { get; init; } // Raw markdown body (below frontmatter)
      public required string SkillDirectory { get; init; } // Absolute path to the skill's directory
      public required string Source { get; init; }  // "user" or "workspace"
  }
  ```

- [x] **Create `src/Ur/Skills/SkillFrontmatter.cs`** -- parses the `---` delimited YAML frontmatter from a SKILL.md file. Returns a `SkillDefinition` given the file content and directory path.
  - Split content on `---` delimiters to extract frontmatter block and body.
  - Use YamlDotNet to deserialize the frontmatter into a dictionary or typed object.
  - Map frontmatter keys (`name`, `description`, `when_to_use`, `arguments`, `argument-hint`, `allowed-tools`, `model`, `user-invocable`, `disable-model-invocation`, `context`, `agent`, `paths`, `version`) to `SkillDefinition` properties.
  - `arguments` is a comma-separated string parsed into `ArgumentNames`.
  - `paths` and `allowed-tools` are comma-separated strings parsed into arrays.
  - If `name` is missing from frontmatter, derive it from the directory name.
  - If `description` is missing, use an empty string.

### Phase 2: Skill loading and registry

- [x] **Create `src/Ur/Skills/SkillLoader.cs`** -- discovers and loads skills from directories.
  - `LoadFromDirectoryAsync(string skillsDir, string source)` scans `skillsDir` for subdirectories containing `SKILL.md`. For each, reads the file and parses it via `SkillFrontmatter`.
  - `LoadAllAsync(string userSkillsDir, string workspaceSkillsDir)` loads from both directories in parallel. Workspace skills take precedence over user skills on name collision (same deduplication strategy as the reference -- last writer wins by source priority).
  - Errors in individual skill files are logged to stderr and skipped (same resilience pattern as extension loading).

- [x] **Create `src/Ur/Skills/SkillRegistry.cs`** -- holds loaded skills and provides lookup.
  ```csharp
  public sealed class SkillRegistry
  {
      // Lookup by name (case-insensitive).
      public SkillDefinition? Get(string name);
      
      // All skills the model can invoke (excludes DisableModelInvocation).
      public IReadOnlyList<SkillDefinition> ModelInvocable();
      
      // All skills the user can invoke via slash commands (excludes !UserInvocable).
      public IReadOnlyList<SkillDefinition> UserInvocable();
      
      // All loaded skills.
      public IReadOnlyList<SkillDefinition> All();
  }
  ```

### Phase 3: Argument and variable substitution

- [x] **Create `src/Ur/Skills/SkillExpander.cs`** -- expands a skill's content for invocation.
  - `Expand(SkillDefinition skill, string args, string sessionId)` returns the final prompt string.
  - Replaces `$ARGUMENTS` with the raw args string.
  - If the skill has `ArgumentNames`, splits `args` by whitespace and replaces `$arg_name` for each named argument. Leftover args go into `$ARGUMENTS`.
  - Replaces `${UR_SKILL_DIR}` with `skill.SkillDirectory`.
  - Replaces `${UR_SESSION_ID}` with `sessionId`.
  - TODO comment: shell execution (`!`command``) for future implementation.

### Phase 4: System prompt and AgentLoop integration

- [x] **Modify `src/Ur/AgentLoop/AgentLoop.cs`** -- add system prompt parameter.
  - Add `string? systemPrompt = null` parameter to `RunTurnAsync`.
  - Before the `while (true)` loop, if `systemPrompt` is not null, create a `new ChatMessage(ChatRole.System, systemPrompt)` and prepend it to the messages list sent to the LLM. Important: prepend to a *copy* or insert-and-remove, so the system message is not persisted to the conversation history. The cleanest approach: build a wrapper list that yields the system message first, then the conversation messages, without mutating `messages`.
  
  ```csharp
  // Build the message list the LLM sees: optional system prompt + conversation.
  // We don't add the system message to `messages` because that list is persisted
  // to disk by UrSession -- the system prompt is transient and rebuilt each turn.
  IEnumerable<ChatMessage> BuildLlmMessages(string? systemPrompt, List<ChatMessage> messages)
  {
      if (systemPrompt is not null)
          yield return new ChatMessage(ChatRole.System, systemPrompt);
      foreach (var msg in messages)
          yield return msg;
  }
  ```

- [x] **Create `src/Ur/Skills/SystemPromptBuilder.cs`** -- constructs the system prompt.
  - `Build(SkillRegistry skills)` returns a string listing available skills for the model.
  - Format mirrors the reference's system-reminder style:
    ```
    The following skills are available for use with the skill tool:
    
    - commit: Use this skill to commit changes to the repository...
    - review-pr: Review a pull request for code quality...
    ```
  - Each skill entry includes: name, description, and `when_to_use` (if present, truncated to ~250 chars).
  - Only includes skills where `DisableModelInvocation` is false.
  - Returns null if no skills are available (so no system message is injected).

### Phase 5: Skill tool (model invocation)

- [x] **Create `src/Ur/Tools/SkillTool.cs`** -- the built-in tool the model calls.
  - Parameters: `skill` (string, required), `args` (string, optional).
  - Tool description (adapted from reference prompt):
    ```
    Execute a skill within the main conversation.
    
    When users ask you to perform tasks, check if any of the available skills match.
    Skills provide specialized capabilities and domain knowledge.
    
    When users reference a "slash command" or "/<something>", they are referring to
    a skill. Use this tool to invoke it.
    
    How to invoke:
    - Use this tool with the skill name and optional arguments
    - Examples:
      - skill: "commit", args: "-m 'Fix bug'"
      - skill: "review-pr", args: "123"
    
    Important:
    - Available skills are listed in the system message
    - When a skill matches the user's request, invoke it immediately
    - Do not invoke a skill that is already running
    ```
  - Validation: strip leading `/` from skill name, look up in `SkillRegistry`. Return error if not found or if `DisableModelInvocation` is true.
  - Execution: call `SkillExpander.Expand()` and return the expanded content as the tool result string.
  - Permission: `ReadInWorkspace` (auto-allowed) -- skill content is just prompt text, not a side-effecting operation.

- [x] **Register Skill tool in `src/Ur/Tools/BuiltinTools.cs`**.
  - Add to `RegisterAll` alongside the existing tools.
  - The `SkillTool` needs access to the `SkillRegistry`, so `RegisterAll` needs to accept it as a parameter (or BuiltinTools grows a second registration method for skill-dependent tools).
  - Preferred: add `RegisterSkillTool(ToolRegistry registry, SkillRegistry skills)` as a separate method called by UrHost after skills are loaded.

### Phase 6: Slash command processing (user invocation)

- [x] **Add slash command handling to `src/Ur/Sessions/UrSession.cs`**.
  - In `RunTurnAsync`, before constructing the user message, check if `userInput` starts with `/`.
  - If so, parse the skill name (first token after `/`) and args (remainder).
  - Look up in `SkillRegistry`. If not found or `UserInvocable` is false, send an error message back (yield an `Error` event and return).
  - If found, expand via `SkillExpander` and replace `userInput` with the expanded content.
  - Wrap the expanded content in tags so the model knows it came from a skill invocation:
    ```
    <command-name>/skill-name</command-name>
    <command-args>the args</command-args>
    
    [expanded skill content]
    ```

- [x] **Wire system prompt into `UrSession.RunTurnAsync`**.
  - After the readiness check, build the system prompt via `SystemPromptBuilder.Build(skillRegistry)`.
  - Pass it to `agentLoop.RunTurnAsync(messages, wrappedCallbacks, systemPrompt, ct)`.

## Gaps and follow-up

- **Conditional skills** (`paths` frontmatter): Parsed and stored on `SkillDefinition` but not activated. Requires hooking into file tool results to activate skills when matching files are touched. TODO comments in `SkillLoader` and `SkillRegistry`.
- **Shell execution** (`!`command``): TODO comment in `SkillExpander`. Requires subprocess execution during expansion, plus security considerations.
- **Allowed-tools / model override**: Parsed on `SkillDefinition` but not enforced. Requires `AgentLoop` to accept tool/model overrides per-turn. TODO comments in `SkillTool`.
- **Forked execution** (`context: fork`): Parsed but always runs inline. Requires a sub-agent mechanism. TODO comment in `SkillTool`.
- **Compaction preservation**: When context compaction is added, invoked skills should be preserved. TODO comment in `UrSession`.

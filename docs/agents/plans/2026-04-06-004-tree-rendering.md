# Tree-Based TUI Rendering

## Goal

Replace the bubble-chrome rendering model (▎ bars, padding rows, background fill) with a tree-drawing model using box-drawing characters (├─, └─, │). The conversation becomes a tree where user messages are roots and all other events (tool calls, assistant messages, subagent blocks) are children.

**Target visual:**

```
❯ This is the user message!
│ If it has to wrap, it wraps like this.
├─ ● read_file(file_path: "bar.txt", offset: "0", limit: "10000")
├─ ● subagent(prompt: "Blah bla...")
│    ├─ ● read_file(file_path: "bar.txt", offset: "0", limit: "2000")
│    └─ ● write_file(file_path: "bar.txt", content: "oof")
├─ ● This is the assistant message!
│    it wraps like this
├─ ● read_file(file_path: "bar.txt", offset: "0", limit: "10000")
└─ ● Another assistant message.

❯ This is the second user message!
├─ ...
```

## Structural considerations

### Tree chrome geometry

Two levels of chrome, measured in columns:

| Context | First row | Continuation | Width |
|---------|-----------|--------------|-------|
| User root | `❯ ` | `│ ` (or `  ` if no children) | 2 |
| Child node | `├─ ● ` or `└─ ● ` | `│    ` or `     ` | 5 |

Content width = `availableWidth - chrome`. No right-pad or right-margin (no background fill in tree mode).

### Grouping model

EventList's flat `_children` list is partitioned into **tree groups** at render time:

- Each `BubbleStyle.User` item starts a new group and becomes the group's root.
- All non-User items following a User item (until the next User or end-of-list) are that root's children.
- Items before the first User item form an "orphan" group (children with no root — e.g., the welcome message).

Within a group, no blank-line separators between siblings. One blank line between groups.

### Last-child detection

`└─` vs `├─` is computed at render time by checking whether a child is the last in its group. During streaming, the currently-last child renders with `└─`; when a new event arrives, it becomes `├─` on the next frame. This is automatic since Render() recalculates every ~33ms.

### SubagentRenderable redesign

Currently renders as a bordered box (header rule + inner EventList + footer rule) with `BubbleStyle.None`. New design:

- Treated as a regular `BubbleStyle.Circle` child in the outer EventList.
- `Render()` returns: row 0 = tool call signature (like ToolRenderable), rows 1+ = inner EventList tree output.
- Outer EventList adds `├─ ● ` to row 0 and `│    ` to rows 1+ (standard child handling).
- Inner EventList renders its own children with tree connectors (all Circle, no User root).
- Header/footer rules and `IndentWidth` are removed entirely.
- Width math: outer chrome (5) + inner tree row (contentWidth) = availableWidth. Inner chrome (5) further narrows content for nested items. Total overhead for doubly-nested content: 10 columns.

### BubbleStyle simplification

Remove `System`, `Assistant`, and `None`. Keep:

- **User** — tree root (❯ glyph)
- **Circle** — tree child (● glyph, color from `Func<Color>?`)

All current callers migrate: `System` → `Circle` with appropriate color, `Assistant` → `Circle`, `None` → `Circle` (SubagentRenderable now renders its own inner tree rows, outer adds tree prefix).

### Subagent tail-clip ellipsis

When the inner EventList exceeds `MaxInnerRows`, older events are dropped. To signal this visually, SubagentRenderable inserts a `├─ ● ...` ellipsis row as the first visible inner row (in dim `BrightBlack`). The ellipsis is always `├─` (never `└─`) because clipped events are by definition not the last child.

```
├─ ● subagent(prompt: "Blah bla...")     ← 5 inner events, all visible
│    ├─ ● read_file(A)
│    ├─ ● read_file(B)
│    ├─ ● read_file(C)
│    ├─ ● read_file(D)
│    └─ ● read_file(E)

├─ ● subagent(prompt: "Blah bla...")     ← 6 inner events, A clipped
│    ├─ ● ...
│    ├─ ● read_file(C)
│    ├─ ● read_file(D)
│    ├─ ● read_file(E)
│    └─ ● read_file(F)
```

The ellipsis row is generated by SubagentRenderable (not the inner EventList) after tail-clipping, so the inner EventList stays generic. The ellipsis row uses the same tree chrome as a regular Circle child — it's just a static `CellRow` with `├─ ● ...` text.

### ToolRenderable cleanup

Remove `→ ok` and `→ error` suffixes from completed state. Circle colors (green/red) convey status. Keep `[awaiting approval]` text (yellow circle alone doesn't distinguish "awaiting" from "running").

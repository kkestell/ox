# PHAME Review 2 — Abstraction and Encapsulation Fixes

## Goal

Address the 14 findings from the second PHAME design review of the Ur codebase.
All changes are pure structural refactoring — no new features, no behavior changes,
no public API surface added.

## Desired outcome

- `AgentLoop.RunTurnAsync` reads as a single-level workflow: no inline tool dispatch,
  no manual `FunctionResultContent` construction, no `errorSink` side-channel visible
  at the call site.
- Lua marshalling lives in its own utility; `LuaToolAdapter` calls it rather than
  implementing it.
- `Extension` exposes a single internal transition point instead of six state-mutating
  methods spread across callers.
- `UrConfiguration` offers typed `GetStringSetting` / `GetBoolSetting` accessors so
  callers never touch `JsonElement`.
- `ModelCatalog` communicates its load state explicitly instead of relying on
  empty `Models` as a sentinel.
- `ToolRegistry` has one `Register` signature; optional parameters replace the
  parallel public/internal overloads.
- All tests pass without behavior changes.

## How we got here

A PHAME review of the full codebase identified 2 critical, 7 high, and 5 medium
findings. Findings cluster around three hot spots: `AgentLoop` (three separate
abstraction violations in one 150-line method), the Extensions subsystem (Lua runtime
lifecycle details leaking through multiple class boundaries), and the
Configuration/Providers layer (JSON and cache-state leaking into callers).

Plan 010 fixed the hierarchy and modularization findings from a prior review. These
findings are the orthogonal abstraction and encapsulation layer.

## Approaches considered

### Option A — Fix in place, finding by finding

Address each of the 14 findings independently, touching only the immediately flagged
lines.

- Pros: Low risk; each change is localized; easy to review incrementally.
- Cons: Several findings share root causes. Fixing AgentLoop in three separate passes
  adds churn; the underlying structure is the same each time. Misses the chance to
  give the fix a clear name and home (e.g., `ToolInvoker`).
- Failure modes: Fixes remain scattered; next review will find the same clusters.

### Option B — Group by structural theme, extract named abstractions

Cluster the findings by the abstraction they need and fix each cluster with a single
well-named extraction. Four themes: AgentLoop dispatch responsibility, Lua marshalling,
Extension state lifecycle, and Configuration/Catalog readability.

- Pros: Each phase has a clear deliverable; extracted types are reusable; the
  resulting code is easier to reason about and test.
- Cons: Slightly more upfront design; two new types (`LuaValueMarshaller`,
  `ToolInvoker`) require their own files.
- Failure modes: New types could be over-engineered; keep them small with no
  public constructors and no dependencies outside their namespace.

## Recommended approach

**Option B.** The structural extractions are small (< 60 lines each), follow existing
patterns in the codebase, and fix multiple review findings in one coherent change.
The `errorSink` finding is a C# language limitation (no yield inside try/catch); the
fix is to rename the side channel to a named struct instead of a raw array, which makes
the intent legible.

## Related code

- `src/Ur/AgentLoop/AgentLoop.cs` — three findings; will gain a `ToolInvoker` helper
- `src/Ur/AgentLoop/ToolRegistry.cs` — parallel Register overloads; unified with optional params
- `src/Ur/Extensions/LuaToolAdapter.cs` — marshalling to extract into `LuaValueMarshaller`
- `src/Ur/Extensions/Extension.cs` — six state-mutating internal methods; collapse to `ApplyState`
- `src/Ur/Extensions/ExtensionLoader.cs` — manifest parsing mixed with orchestration
- `src/Ur/Configuration/UrConfiguration.cs` — `GetSetting`/`SetSettingAsync` expose `JsonElement`
- `src/Ur/Providers/ModelCatalog.cs` — `LoadCache` returns bool with no state property
- `src/Ur/Permissions/PermissionGrantStore.cs` — lazy-load strategy exposed by public `EnsureX` calls
- `src/Ur/Skills/SkillExpander.cs` — string manipulation mixed with template expansion logic
- `src/Ur/Skills/SkillFrontmatter.cs` — inline string manipulation inside `ParseFrontmatter`
- `src/Ur/Tools/ToolArgHelpers.cs` — `ExtractStringArg` takes raw `IDictionary` instead of typed args
- `src/Ur/Sessions/UrSession.cs` — `RunTurnAsync` mixed levels (already improved by plan 010 but residual)
- `tests/Ur.Tests/` — all tests must pass after each phase

## Current state

- `AgentLoop.RunTurnAsync` already has `BuildLlmMessages` and `StreamLlmAsync` extracted.
  The `errorSink` array is a documented workaround for the C# yield/try/catch constraint.
  The 40-line tool dispatch block (permission check → lookup → invoke → result) is still
  inline and is the primary remaining concern.
- `ToolRegistry` has two `Register` overloads: `public Register(AIFunction)` and
  `internal Register(AIFunction, OperationType, ...)`. The public overload defaults to
  `WriteInWorkspace`, the internal one is explicit. The duplication is the issue, not the
  access levels.
- `Extension` already marks `LuaState` as `internal` and has internal lifecycle methods.
  The concern is that `ExtensionLoader` calls five of them directly instead of going
  through a single transition point.
- `UrConfiguration.GetSetting` returns `JsonElement?` and `SetSettingAsync` takes
  `JsonElement`. All callers that set model ID already route through typed methods
  (`SetSelectedModelAsync`). The gap is the generic extension-settings path.
- `ModelCatalog.LoadCache()` returns `bool` (did the cache load?). `Models` is empty
  before a load succeeds, which forces callers to treat emptiness as a sentinel.
- `PermissionGrantStore.IsCovered` calls two private `EnsureX` methods synchronously
  doing file I/O. These are private — the finding is about the design smell of
  synchronous lazy file I/O in a public method, not exposed API surface.
- `ToolArgHelpers.ExtractStringArg` takes `IDictionary<string, object?>` because
  `PermissionMeta.TargetExtractor` is `Func<IDictionary<string, object?>, string>`.
  If `TargetExtractor` took `AIFunctionArguments` instead, `ExtractStringArg` could
  be removed (the typed helpers already exist).

## Structural considerations

**Hierarchy**: None of the proposed changes introduce new cross-layer dependencies.
`LuaValueMarshaller` stays in `Ur.Extensions`; a notional `ToolInvoker` stays in
`Ur.AgentLoop`. No new inter-namespace edges.

**Abstraction**: The main goal. Each extraction reduces a method that mixes two
levels (dispatch + invocation, or parsing + string manipulation) to one that operates
at a single level.

**Modularization**: `LuaValueMarshaller` is a genuine utility (reusable if a second
Lua adapter is ever needed). Other extractions are private helpers within existing
classes — no nano-modules.

**Encapsulation**: `Extension` internal state becomes less visible to `ExtensionLoader`;
`ModelCatalog` state becomes less ambiguous; `PermissionGrantStore` lazy-load timing
becomes an implementation detail rather than a caller concern.

## Implementation plan

### Phase 1 — AgentLoop cleanup (critical + high)

- [x] **1a. ToolInvoker.** Create `src/Ur/AgentLoop/ToolInvoker.cs` (internal, sealed).
  Move the `foreach (var call in toolCalls)` block from `RunTurnAsync` into
  `ToolInvoker.InvokeAllAsync`. The method signature:
  ```
  internal IAsyncEnumerable<AgentLoopEvent> InvokeAllAsync(
      IReadOnlyList<FunctionCallContent> calls,
      ChatMessage resultMessage,
      TurnCallbacks? callbacks,
      CancellationToken ct)
  ```
  `ToolInvoker` is constructed with the `ToolRegistry`. It emits `ToolCallStarted`
  and `ToolCallCompleted` events and appends `FunctionResultContent` items to
  `resultMessage` — callers only need to add `resultMessage` to `messages` afterward.
  Permission checking moves into `ToolInvoker` (it already has access to the registry
  for metadata lookup). Remove `IsPermissionDeniedAsync` from `AgentLoop`; pass the
  check or factor it into `ToolInvoker.CheckPermissionAsync` (private).

- [x] **1b. errorSink naming.** Replace `Exception?[1]` array with a local
  `LlmError` record: `record struct LlmError(Exception? Exception)`. Update
  `StreamLlmAsync` signature to `... LlmError error` (ref parameter or tuple return).
  The comment explaining the C# constraint stays — the fix is legibility, not removal.
  Alternative simpler fix: use a single-element `StrongBox<Exception?>` from
  `System.Runtime` if a new record feels heavy. Either way, document the constraint
  inline so future readers don't wonder why it exists.
  _Note_: If the `Exception?[1]` pattern already reads clearly with its existing
  comment, a rename to `errorHolder` with a doc comment is acceptable as a
  minimum-viable fix so effort goes to higher-value changes.

- [x] **1c. RunTurnAsync audit.** After 1a, verify `RunTurnAsync` reads as a
  pure loop: `BuildLlmMessages → stream → optionally yield tool events → persist`.
  No `new ChatMessage(...)` with manual content manipulation should remain inline;
  extract any residual message construction into named builder methods.

### Phase 2 — Extensions subsystem

- [x] **2a. LuaValueMarshaller.** Create `src/Ur/Extensions/LuaValueMarshaller.cs`
  (internal static class). Move the four private static methods
  (`MarshalToLua`, `MarshalJsonElementToLua`, `MarshalJsonObjectToLuaTable`,
  `MarshalJsonArrayToLuaTable`) from `LuaToolAdapter` into it as internal static
  methods (exact same logic, new home). Change the two calls in
  `LuaToolAdapter.InvokeCoreAsync` to `LuaValueMarshaller.ToLua(value)` and
  `LuaValueMarshaller.FromLua(results[0])`. Also add `FromLua` (the reverse
  direction currently implemented inline in `InvokeCoreAsync`) for symmetry and
  future use.

- [x] **2b. Extension ApplyState.** Replace the five internal lifecycle methods
  (`Activate`, `MarkActivated`, `MarkActivationFailed`, `MarkDeactivated`,
  `ResetRuntimeState`) with a single `internal void ApplyState(ExtensionState state,
  LuaState? lua = null, string? error = null)` method where `ExtensionState` is a
  private enum (`{ Inactive, Active, Failed }`). Move the state transition logic
  (set `IsActive`, `LuaState`, `LoadError`, clear `_tools`) into `ApplyState`.
  Update `ExtensionLoader` call sites to use the new method.
  _Constraint_: `RegisterTool` stays separate — it's a distinct operation (adding a
  tool to an active extension) not a lifecycle transition.

- [x] **2c. ManifestParser.** Extract private methods from `ExtensionLoader` that
  do pure Lua-table-to-C#-domain translation into an internal static class
  `ManifestParser` in `src/Ur/Extensions/ManifestParser.cs`. The manifest evaluation
  step (running the Lua script) stays in `ExtensionLoader`; `ManifestParser` receives
  the `LuaTable` result and returns an `ExtensionDescriptor`. This separates Lua I/O
  from Lua domain mapping.

### Phase 3 — Configuration and Providers

- [x] **3a. UrConfiguration typed accessors.** Add typed helper methods alongside
  the existing generic pair:
  ```csharp
  public string? GetStringSetting(string key)
  public bool? GetBoolSetting(string key)
  public Task SetStringSettingAsync(string key, string value,
      ConfigurationScope scope = ConfigurationScope.User, CancellationToken ct = default)
  public Task SetBoolSettingAsync(string key, bool value,
      ConfigurationScope scope = ConfigurationScope.User, CancellationToken ct = default)
  ```
  These wrap `GetSetting`/`SetSettingAsync` with `JsonSerializer.Deserialize` /
  `JsonSerializer.SerializeToElement` internally. Keep the generic `GetSetting` /
  `SetSettingAsync` as `internal` if no external callers remain, or at minimum keep
  them but add `[EditorBrowsable(Never)]` to steer callers toward the typed accessors.
  Update known call sites (extension settings in `UrConfiguration.cs` itself and any
  CLI/TUI code).

- [x] **3b. ModelCatalog state.** Add a `CatalogState` enum (`{ Empty, Loaded,
  LoadFailed }`) and a `public CatalogState State { get; private set; }` property.
  Set `State = CatalogState.Loaded` on successful load/refresh, `LoadFailed` on
  `JsonException` catch in `LoadCache`. Update callers (primarily `UrHost.StartAsync`)
  to check `catalog.State == CatalogState.Loaded` instead of `Models.Count > 0`.
  `IsReady` can be a convenience property: `public bool IsReady => State == CatalogState.Loaded`.

### Phase 4 — Polish and medium findings

- [x] **4a. ToolRegistry unified Register.** Remove the public no-metadata `Register`
  overload. Give the internal overload default values (`OperationType operationType =
  OperationType.WriteInWorkspace, string? extensionId = null, Func<...>?
  targetExtractor = null`) and make it `public`. Callers that registered with default
  behavior now use the single overload with only `AIFunction tool` specified
  (by position). Verify all call sites compile.

- [x] **4b. PermissionGrantStore lazy-load intent.** The `EnsureX` methods are already
  private — no public surface to remove. The actionable fix is to document the
  synchronous lazy-load trade-off in the class-level XML comment (why eagerly loading
  on construction is not done: `PermissionGrantStore` is created synchronously in
  `UrSession`'s constructor, and an async factory would need to propagate through the
  session stack). If the constructor can be changed to an async factory without
  cascading changes, do so; otherwise document the constraint and annotate
  `IsCovered` with a note about synchronous file I/O.

- [x] **4c. TargetExtractor typed argument.** Change `PermissionMeta.TargetExtractor`
  from `Func<IDictionary<string, object?>, string>` to `Func<AIFunctionArguments,
  string>`. Update all lambdas in `BuiltinTools.cs` that call
  `ToolArgHelpers.ExtractStringArg(args, key)` to call
  `ToolArgHelpers.GetOptionalString(args, key) ?? "(unknown)"` directly (the existing
  typed helper already handles both string and JsonElement). Remove
  `ToolArgHelpers.ExtractStringArg` once no callers remain.

- [x] **4d. SkillFrontmatter yaml helpers.** The parser is already split into
  `ParseFrontmatter`, `SplitFrontmatter`, `UnquoteYaml`, and `ParseBool`. The
  remaining "mixed level" is the `foreach (var rawLine in yaml.Split('\n'))` loop
  in `ParseFrontmatter` that does key-value extraction inline. Extract a private
  `static IEnumerable<(string Key, string Value)> ParseKeyValues(string yaml)` method
  that handles the `Split`, `Trim`, colon-splitting, and `UnquoteYaml` call, so that
  `ParseFrontmatter` iterates clean key-value pairs and only switches on `key`.

- [x] **4e. SkillExpander template method.** Extract the four `content.Replace(...)`
  calls at the bottom of `Expand` into a single `private static string ApplyBuiltins(
  string content, string remainingArgs, string skillDirectory, string sessionId)`
  method. The named-argument substitution loop stays in `Expand`. No new class needed.

## Impact assessment

- Code paths affected: `AgentLoop`, `Extensions` subsystem, `UrConfiguration`,
  `ModelCatalog`, `PermissionGrantStore`, `SkillExpander`, `SkillFrontmatter`,
  `ToolRegistry`, `ToolArgHelpers`/`BuiltinTools`.
- Data or schema impact: None — all behavioral logic unchanged.
- Dependency or API impact: `UrConfiguration` gains new typed public methods;
  `ToolRegistry.Register` signature changes (but callers inside the assembly
  compile unchanged due to optional params). `ModelCatalog.State` is new public
  surface.

## Validation

- Run `dotnet test` after each phase; all tests must pass before starting the next.
- After Phase 1, manually trace through `AgentLoop.RunTurnAsync` and confirm it
  reads as a single-level loop with no inline data transformation.
- After Phase 4c, confirm `ToolArgHelpers.ExtractStringArg` has been removed and
  grep finds zero references.
- Run `make inspect` and verify `inspection-results.txt` has no new issues.

# Event-Driven Architecture for Te + Ox

## Context

The current rendering and input systems are built on wasteful polling patterns:
- A 30fps `System.Threading.Timer` fires constantly even when nothing changes
- `InputReader.ReadLineInViewport` polls with `Thread.Sleep(20)` to check for input
- Escape key during turns uses a split input path (direct `inputSource.KeyPressed` subscription bypassing the coordinator)
- Three threads (stdin reader, main polling loop, timer callback) do a careful dance that's fragile and hard to reason about

The previous plan tried to patch these problems incrementally (add `SemaphoreSlim` to the coordinator, make `ReadLineInViewport` async). That approach was wrong — it bolted async onto a fundamentally polling-based design without questioning the design itself.

This plan replaces the polling model with an event-driven architecture: render only when something changes, wake only when input arrives, use one unified input path.

## Design Principles

1. **Render only when dirty.** No periodic timer. The only timer is for the throbber, scoped to turns only.
2. **No `Thread.Sleep` anywhere.** Input reading genuinely awaits input via `Channel<T>`.
3. **Single input path.** All keys flow through `InputCoordinator`. No direct `inputSource.KeyPressed` subscriptions in Ox.
4. **Keep the background stdin reader thread** — unavoidable for Unix raw mode.
5. **Keep `ConsoleBuffer`'s dirty-cell tracking** — already correct and efficient.

## Architecture

### Threading model (after)

1. **Background stdin reader thread** (in `TerminalInputSource`): reads bytes, parses ANSI, calls `coordinator.Enqueue()` which writes to a channel AND fires events eagerly on the calling thread.
2. **Main loop thread** (`TuiService.ExecuteAsync`): during input reading, awaits the coordinator's channel; during turns, `await foreach`es model events. Calls `viewport.RedrawIfDirty()` after processing events.
3. **Throbber timer thread** (ThreadPool, only during turns): calls `viewport.ThrobberTick()` → `Redraw()` under `_redrawLock`. 1-second period.

### Key design decisions

**InputCoordinator fires events eagerly on enqueue.** Currently events only fire during `ProcessPendingInput()`. New behavior: `Enqueue()` writes to the channel AND fires `KeyReceived`/`MouseReceived` immediately (on the background stdin thread). This enables escape detection during turns without polling — subscribe to `coordinator.KeyReceived`, which fires instantly when a key is enqueued.

**InputReader reads from the channel, not events.** `ReadLineInViewportAsync` reads from `coordinator.Reader` (the `ChannelReader<InputEvent>`), processing keys serially on the main thread. It does NOT subscribe to `KeyReceived` (which fires on the background thread). This avoids cross-thread mutation of the input buffer.

**Viewport exposes `RedrawIfDirty()` instead of running a timer.** The main loop calls this at explicit points. The `_dirty` volatile bool is set by `Changed` event handlers (from any thread), but `RedrawIfDirty()` checks and clears it under `_redrawLock`. The throbber timer also calls `RedrawIfDirty()` — the lock serializes access.

**During input reading, only input triggers redraws.** No renderables change during input reading (streaming only happens during turns, sidebar only updates after turns). So there's no need for a separate dirty-wakeup mechanism. `ReadLineInViewportAsync` calls a redraw callback after processing each key.

## Implementation plan

### Phase 1: InputCoordinator — Channel + eager events

Replace the `ConcurrentQueue<InputEvent>` with `Channel<InputEvent>`. Fire events eagerly on enqueue. Expose the channel reader.

**`src/Te/Input/InputCoordinator.cs`:**
- Replace `private readonly ConcurrentQueue<InputEvent> _pending` with `private readonly Channel<InputEvent> _channel = Channel.CreateUnbounded<InputEvent>()`
- Add `public ChannelReader<InputEvent> Reader => _channel.Reader`
- Change `Enqueue()` to write to `_channel.Writer.TryWrite()` AND fire `InputReceived`/`KeyReceived`/`MouseReceived` immediately
- Remove `ProcessPendingInput()` entirely
- Remove `PendingInputCount` (or reimplement via `_channel.Reader.CanCount ? _channel.Reader.Count : 0` if needed for tests)
- In `Dispose()`, also call `_channel.Writer.TryComplete()` so any pending `WaitToReadAsync` unblocks

**`tests/Te.Tests/InputCoordinatorTests.cs`:**
- `ProcessPendingInput_PreservesArrivalOrderAcrossInputKinds` — events now fire eagerly on enqueue, so the assertion changes: events are observed immediately after `EmitKey`/`EmitMouse`, not after calling `ProcessPendingInput`. Or, verify ordering via channel reads instead.

**`src/Te/Program.cs` (demo):**
- Replace `coordinator.ProcessPendingInput()` + `Thread.Sleep(16)` with reading from `coordinator.Reader`. The demo subscribes to `KeyReceived`/`MouseReceived` which now fire eagerly, so it can just await the channel for wake-up and redraw. Or keep it simple: subscribe to events (already works since they fire eagerly), use `await coordinator.Reader.WaitToReadAsync()` as the sleep replacement, drain to keep the channel from growing.

### Phase 2: InputReader — async, no Sleep

**`src/Ox/InputReader.cs`:**
- Rename `ReadLineInViewport` to `ReadLineInViewportAsync`, return `async Task<string?>`
- Add `Action? onInputProcessed` parameter (called after each key, so caller can trigger redraw)
- Replace the inner loop:
  ```csharp
  // Old: subscribe to KeyReceived, poll with Thread.Sleep
  // New: read from channel directly
  while (!ct.IsCancellationRequested && !done)
  {
      await _coordinator.Reader.WaitToReadAsync(ct);
      while (_coordinator.Reader.TryRead(out var inputEvent))
      {
          if (inputEvent is KeyInputEvent keyInput)
              OnKey(keyInput.Key);
          if (done) break;
      }
      onInputProcessed?.Invoke();
  }
  ```
- Remove the `_coordinator.KeyReceived += OnKey` / `-= OnKey` pattern — read from channel instead
- Similarly convert `ReadLine` to `ReadLineAsync`
- Delete all `Thread.Sleep` calls

### Phase 3: Viewport — remove timer, add RedrawIfDirty

**`src/Ox/Rendering/Viewport.cs`:**
- Remove `_redrawTimer` field and its initialization in the constructor
- Remove `TickRedraw()` method
- Add public method:
  ```csharp
  public void RedrawIfDirty()
  {
      if (_dirty && _running)
          Redraw();
  }
  ```
- Extract throbber advancement into a public method:
  ```csharp
  public void ThrobberTick()
  {
      if (!_turnRunning) return;
      var counter = GetCurrentThrobberCounter();
      if (counter != _lastAnimatedThrobberCounter)
      {
          _lastAnimatedThrobberCounter = counter;
          Redraw(); // Under _redrawLock, directly
      }
  }
  ```
- `Dispose()` — remove `_redrawTimer.Dispose()` (no timer to dispose)
- Keep `_dirty`, `_redrawLock`, `Redraw()`, `BuildFrame()` unchanged
- Keep `Changed` handlers setting `_dirty = true` (they still work — the main loop polls `RedrawIfDirty` after events)

### Phase 4: Program.cs — rewire the main loop

**`src/Ox/Program.cs`:**

Input reading phase:
- Change `inputReader.ReadLineInViewport(...)` to `await inputReader.ReadLineInViewportAsync(...)` with `viewport.RedrawIfDirty` as the `onInputProcessed` callback

Turn execution phase:
- After each `router.RouteMainEvent(evt)`, call `viewport.RedrawIfDirty()`
- Create a per-turn `Timer(1000, 1000)` calling `viewport.ThrobberTick()` — disposed in the finally block
- Subscribe to `coordinator.KeyReceived` for escape detection (replaces `inputSource.KeyPressed` subscription). Add a `_permissionPromptActive` guard so Escape during permission prompts doesn't cancel the turn.
- Remove the direct `inputSource.KeyPressed += escapeHandler` subscription entirely

Cleanup:
- Call `viewport.RedrawIfDirty()` after `viewport.SetTurnRunning(false)` to ensure the throbber clears

### Phase 5: PermissionHandler — async

**`src/Ox/PermissionHandler.cs`:**
- Change `inputReader.ReadLineInViewport(...)` to `await inputReader.ReadLineInViewportAsync(...)` with `viewport.SetInputPrompt` as prompt callback and `viewport.RedrawIfDirty` as input-processed callback
- The `RequestPermissionAsync` lambda becomes truly async (it already returns `ValueTask<PermissionResponse>`)

### Phase 6: EnsureReadyAsync — async

**`src/Ox/Program.cs` `EnsureReadyAsync`:**
- Change `inputReader.ReadLine(ct)` to `await inputReader.ReadLineAsync(ct)`

## What stays unchanged

- `TerminalInputSource` — background stdin thread, `ReadByteWithTimeout`, `ReadUnixLoop`. Untouched.
- `ConsoleBuffer` — dirty-cell tracking, double buffering. Untouched.
- `IRenderable`, `TextRenderable`, `ToolRenderable`, `SubagentRenderable`, `EventList` — fire `Changed` events as before.
- `EventRouter` — routes events to renderables. Untouched.
- `Sidebar`, `ContextSection`, `TodoSection` — untouched.
- `tests/Ur.Tests/TuiRenderingTests.cs` — no references to timer/polling infrastructure.

## What gets deleted

- The 33ms `Timer` in `Viewport` constructor
- `TickRedraw()` in `Viewport`
- `ProcessPendingInput()` in `InputCoordinator`
- `PendingInputCount` in `InputCoordinator` (or kept as channel reader count)
- All `Thread.Sleep()` calls in `InputReader`
- `Thread.Sleep(16)` in `Te/Program.cs` demo
- Direct `inputSource.KeyPressed += escapeHandler` in `Program.cs`
- `ConcurrentQueue<InputEvent>` import in `InputCoordinator`

## Verification

After each phase: `dotnet build` must pass with zero errors.

After Phase 4 (full integration): manual smoke test:
- Launch the app — splash screen renders immediately
- Type characters — input appears with no perceptible latency
- Press Tab — autocomplete works
- Submit a message — turn runs, streaming text appears, throbber animates at ~1s intervals
- Press Escape mid-turn — turn cancels, `[cancelled]` appears
- Press Ctrl+C — clean exit, terminal restored
- Trigger a permission prompt — prompt appears, responds to y/n, Escape doesn't cancel the turn
- Observe CPU usage at idle (during input reading) — should be near zero, not 3-5% from polling
